# Contributing to outtake

Issues, discussions and pull requests are all welcome — this project aims
to stay small, zero-dependency at runtime, and honest about what a Takeout
export contains.

## Getting started

Requirements: Node.js >= 22.13 (for the stable `node:test` runner used by the suite).

```bash
git clone https://github.com/JaydenCJ/outtake.git
cd outtake
npm install            # installs typescript, the only devDependency
npm run build          # compile TypeScript to dist/
npm test               # build + 92 node:test tests
bash scripts/smoke.sh  # end-to-end CLI check against a generated sample export
```

`scripts/smoke.sh` exercises the real CLI (scan, plan, products, exit
codes, --strict, JSON output, missing-part detection, extracted-tree
scanning, determinism) against a sample export generated by
`examples/make-sample.mjs` and must print `SMOKE OK`.

## Before you open a pull request

1. `npx tsc -p tsconfig.json --noEmit` — the tree must type-check clean (strict mode is enforced).
2. `npm test` — all tests must pass.
3. `bash scripts/smoke.sh` — must print `SMOKE OK`.
4. Add tests for behavior changes; keep logic in pure, unit-testable
   modules (parsing, aggregation and rendering all take values, not file
   handles — only the CLI and the source loader touch the filesystem).
5. Catalog changes (`src/products.ts`) need a source you can cite for the
   folder name and formats — a real export's layout, or Google's own
   per-product Takeout documentation.

## Ground rules

- **No runtime dependencies.** The zero-dependency install is a core
  feature; adding one needs justification in the PR and will usually be
  declined.
- **File data is never read.** outtake lists archives from their central
  directory / headers only — it must stay fast and safe on 50 GB inputs.
  Never add code that inflates user files or opens their contents.
- No network calls, ever — the tool reads local files and prints. That is
  the whole I/O surface. Nothing about the user's export leaves the machine.
- Product ids (`photos`, `mail`, …) are stable API: never rename or
  repurpose an existing id; add aliases instead.
- Next-step suggestions must name real, maintained tools and be honest
  about friction (a converter that "mostly works" should say so).
- Code comments and doc comments are written in English.

## Reporting bugs

Please include: `outtake --version` output, the exact command line, and
the `scan --format json` output if you can share it (it contains file
names and sizes, nothing else). For archive-parsing bugs, the output of
`unzip -l` or `tar -tzf` on the same file is the fastest way to show the
disagreement — never upload the archive itself.

## Security

Do not open public issues for security problems; use GitHub private
vulnerability reporting on this repository instead.
